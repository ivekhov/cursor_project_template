# Cursor: лучшие практики и инструменты



## Структура проекта и требования к архитектуре


Описаны в .cursor/rules внутри проекта, в котором идет разработка с помощью cursor.

Структуру проекта с файлами сделать самостоятельно или попросить агента в режиме Plan спроектировать архитектуру планируемого проекта

**Простой Пример**
```
project/
├── .cursor/rules    # настройки проекта для Курсора
├── TASK.md          # Текущая задача
├── BACKLOG.md       # Список будущих задач
└── DONE.md          # Завершенные задачи
```

## Как формулировать задачи: 

### Задача средняя (30 минут - 2 часа): через описание в файле и чат с агентом

Если задачу сложно уместить в 3-4 абзаца чата  → создавайте файл и ссылайтесь на него в чате
```Чат: "Прочитай файл TASK.md и реализуй описанную задачу. Начни с backend endpoints."```

### Задача большая (2 часа+): через детальное описание в файле и поэтапно в чате
```
- Чат: "Прочитай docs/FEATURE_NAME.md и реализуй Phase 1" -> [агент делает]
- Чат: "Отлично! Теперь Phase 2" -> [агент делает]
```

### Пример Практического workflow на этом примере

**Утро:**
```markdown
Чат: "Покажи статус задач из BACKLOG.md и предложи что взять сегодня"
```
**В процессе работы:**
```markdown
Чат: "Реализуй пункт 1 из TASK.md"
Чат: "Отметь как выполненное в TASK.md"
Чат: "Переходим к пункту 2"
```
**Конец дня:**
```markdown
Чат: "Обнови TASK.md текущим статусом и перечисли что сделано"
```

## Rules - правила проекта (архитектура, стек, нейминг и тд)

https://cursor.com/ru/docs/context/rules

Работают и читаются всегда, в любом чате с агентом

По сравнению с Skills - Skills  только когда релевантны и касаются специфического контекста

### Проектные правила: размещаются в .cursor/rules 

- Формат: файлы .mdc
- Для проекта пример: .cursor/rules/rule-name.mdc
- .cursorrules - устаревший формат

### Простые и общие правила AGENTS.md 

Для более простых и общих правил в корне проекта файл AGENTS.md 
Это обычный markdown-файл без метаданных и сложных конфигураций. Он идеально подходит для проектов, которым нужны простые, легко читаемые инструкции без дополнительных сложностей структурированных правил.


```md
# Project Instructions

## Code Style
- Use TypeScript for all new files
- Prefer functional components in React
- Use snake_case for database columns

## Architecture
- Follow the repository pattern
- Keep business logic in service layers
```



### Пользовательские правила 

Пользовательские правила — это глобальные настройки, заданные в Cursor Settings → Rules, которые применяются ко всем проектам. Они используются агентом (Chat) и идеально подходят для указания предпочтительного стиля общения или соглашений по оформлению кода:

```
Отвечайте кратко. Избегайте повторений и лишних слов.
```


## Commands - команды

https://cursor.com/ru/docs/context/commands

- Формат: файлы  .md
- Команды проекта: хранятся в каталоге .cursor/commands вашего проекта
- Глобальные команды: хранятся в каталоге ~/.cursor/commands в вашем домашнем каталоге
- Использование: `/command-name`

Когда вы вводите / в поле ввода чата, Cursor автоматически находит и показывает доступные команды из всех этих источников, делая их мгновенно доступными в вашем рабочем процессе.


Пример описания команды в .cursor/commands/code-review-checklist.md: 

```md
# Чек-лист ревью кода

## Обзор
Полный чек-лист для проведения тщательного ревью кода с целью обеспечения качества, безопасности и поддерживаемости.

## Категории ревью

### Функциональность
- [ ] Код делает то, что должен
- [ ] Граничные случаи обработаны
- [ ] Обработка ошибок реализована корректно
- [ ] Нет явных багов или логических ошибок

### Качество кода
- [ ] Код читаем и хорошо структурирован
- [ ] Функции компактны и сфокусированы
- [ ] Имена переменных описательны
- [ ] Нет дублирования кода
- [ ] Соблюдены соглашения проекта

### Безопасность
- [ ] Нет явных уязвимостей безопасности
- [ ] Присутствует валидация входных данных
- [ ] Конфиденциальные данные обрабатываются корректно
- [ ] Нет захардкоженных секретов

```




## Skills - навыки агентов

https://cursor.com/ru/docs/context/skills

Skills — это переиспользуемые пакеты с инструкциями для агента. 

Каждый навык должен быть каталогом, содержащим файл SKILL.md:

```
.agents/
└── skills/
    └── my-skill/
        └── SKILL.md
```

- Если в проекте есть не только агенты Cursor, то .agents/skills/my-skill/SKILL.md
- Если проект только в Cursor, то .cursor/skills/my-skill/SKILL.md
- Использование: вручную `/skill-name`
- Использование: в фоне, когда агент сам решает, что данный скилл стоит использовать


Если в SKILL.md задан параметр disable-model-invocation: true, skill работает только через явный вызов с / и не применяется агентом автоматически.


Агент будет автоматически применять её по контексту. Вызвать вручную можно через /название-навыка в чате (аналог в Claude Code `/COMMANDNAME`)
Навыки также можно вызывать вручную, набрав / в чате Agent и выполнив поиск по имени навыка.

```
---
name: my-skill
description: Short description of what this skill does and when to use it.
---

# My Skill

Detailed instructions for the agent.

## When to Use

- Use this skill when...
- This skill is helpful for...

## Instructions

- Step-by-step guidance for the agent
- Domain-specific conventions
- Best practices and patterns
- Use the ask questions tool if you need to clarify requirements with the user
```

Навыки могут включать каталог scripts/ с исполняемым кодом, который агенты могут запускать. Ссылайтесь на скрипты в файле SKILL.md, используя относительные пути от корня навыка.


Старайтесь делать основной файл SKILL.md компактным и выносите подробные справочные материалы в отдельные файлы. Это повышает эффективность использования контекста, поскольку агенты загружают ресурсы постепенно — только при необходимости.

Миграция правил и команд в навыки
В версии 2.4 Cursor включает встроенный навык /migrate-to-skills, который помогает преобразовать существующие динамические правила и слэш-команды в навыки.




## MCP сервера 

- https://cursor.com/ru/learn/tool-calling 
- https://cursor.com/ru/docs/context/mcp/directory 

MCP-сервер позволяет агенту работать с инструментом (субд / внешним сервисом) и выполнять команды, поставленные в чате, 
которые касаются этих инструментов. Типовой пример - запрос на естественном языке по аналитике данных в БД вида "посчитай мне то-то", "найди в бд записи такие-то"

Подключение через ide или .cursor/mcp.json  

Для внешних интеграций используйте MCP, для переиспользуемых рабочих процессов — Agent Skills, а встроенные инструменты (файлы, терминал, поиск) работают автоматически в режиме агента.

## SubAgents

https://cursor.com/ru/docs/context/subagents


Когда Agent получает сложную задачу, он может автоматически запустить субагента. Субагент получает запрос со всем необходимым контекстом, работает автономно и возвращает итоговое сообщение с результатами.

- В проекте: .cursor/agents/ 
- Каждый субагент — это markdown‑файл с YAML‑фронтматтером:

```md
---
name: security-auditor
description: Специалист по безопасности. Используйте при реализации аутентификации, платежей или обработке конфиденциальных данных.
model: inherit
---

Вы эксперт по безопасности, проводящий аудит кода на наличие уязвимостей.

При вызове:
1. Определите критичные для безопасности участки кода
2. Проверьте наличие распространённых уязвимостей (инъекции, XSS, обход аутентификации)
3. Убедитесь, что секреты не захардкожены
4. Проверьте валидацию и санитизацию входных данных

Сообщайте о находках по степени критичности:
- Критичная (необходимо исправить перед развёртыванием)
- Высокая (исправить в ближайшее время)
- Средняя (устранить при возможности)

```

Явный вызов:  Запросите конкретного субагента, используя синтаксис /name в своём запросе:
```> /security-auditor review the payment module```

Запускайте несколько субагентов одновременно для максимальной производительности:

Субагенты можно возобновлять, чтобы продолжать предыдущие диалоги. Это полезно для длительных задач, которые охватывают несколько вызовов.

Каждый запуск субагента возвращает идентификатор агента. Передайте этот идентификатор, чтобы возобновить работу субагента с полным сохранением контекста

Распространённые шаблоны
- Агент верификации 
- Паттерн оркестрации


Примеры субагентов
- Отладчик
- Тест-раннер

Практики:
- Создавайте узкоспециализированных сабагентов — у каждого сабагента должна быть одна чёткая зона ответственности. Избегайте универсальных «helper»-агентов.
- Уделяйте внимание описаниям — поле description определяет, когда Agent делегирует задачу вашему сабагенту. Потратьте время на его доработку. Тестируйте, создавая промпты и проверяя, что срабатывает нужный сабагент.
- Делайте промпты лаконичными — длинные, сбивчивые промпты размывают фокус. Будьте конкретны и прямы.
- Добавьте сабагентов в систему контроля версий — добавьте .cursor/agents/ в репозиторий, чтобы этим могла пользоваться вся команда.
- Начинайте с агентов, сгенерированных Agent — позвольте Agent помочь с черновой конфигурацией, а затем доработайте её под себя.



## Упоминания

https://cursor.com/ru/docs/context/mentions

Ссылаться на целые файлы можно, выбрав @Files & Folders, а затем указав имя нужного файла для поиска. Вы также можете перетащить файлы из боковой панели прямо в Agent, чтобы добавить их в качестве контекста.

После выбора папки введите /, чтобы перейти внутрь и увидеть все вложенные папки.

Ссылайтесь на конкретные участки кода с помощью символа @Code. Это даёт более детальный контроль по сравнению с @Files & Folders, позволяя выбирать точные фрагменты кода вместо целых файлов.

Функция @Docs позволяет использовать документацию, чтобы проще писать код. В Cursor уже включена популярная документация, и вы можете добавить свою.
